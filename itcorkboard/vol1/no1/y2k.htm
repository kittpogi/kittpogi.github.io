... The Millenium Bug

If a software or an equipment that has/uses software that does not have this bug, it is said to be Y2K-compliant (or, for some, Y2K-ready). In other words, they will continue to function as they do now without any revision or date-related problem on and beyond January 1, 2000.
  Software applications

The bug is related to date translations and date-related operations or computations. There are many programs that use a two-digit year (e.g., 98 for 1998). When 2000 comes, the program will display it as 00. Now, when perform this date-related operation (say, to compute a person’s age)


2000 – 1970 = 30
But using the two-digit year, this is how it will be computed.

00 – 70 = – 70
Actually, some software packages have a certain behavior in translating dates. For example, in Microsoft Excel 97

Notice how it translates 1/1/30 to January 1, 1930 and 1/1/29 to January 1, 2029. This is its 100-year window – i.e., from January 1, 1930 to December 31, 2029. It is how it translates two-digit years.

It appears that about 70% of available dates in the 100-year window is before the current date. About 30% is after the current date. Hence, some call it as the 70/30 rule. Some use 80/20. There is also a problem in Unix operating systems and some programs written in C (a programming language). There is a program variable called time_t, stored in seconds from January 1, 1970. It gives a maximum date value of January 20, 2038. Beyond that, the computer clock will interpret this time as January 1, 1970.

One common problem is the recognition of 2000 as a leap year. Since a year is 365.242 days long, there is a need to make adjustments. One is the leap year. On years evenly divisible by four (4), we have February 29. However, centuries like 1900 and 2000 are leap years only if they are evenly divisible by 400. So, 1700, 1800, and 1900 are not leap years. But 2000 is.

[Subject for verification] Another possible problem is mentioned in ABB India (one of India’s newspaper). It quoted Nehru planetarium director, Dr. J. J. Rawal, confirming that February 2000 will have 30 days. The Gregorian calendar has to be corrected every 400 years.
  Solutions
 

For most software packages that have this problem, the people using them have a few choices.

  1. Wait for the next version and hope that it is Y2K-compliant.
  2. Use another program, which is Y2K-compliant and developed by another vendor.
  3. Develop a Y2K-compliant program to replace the old one.

We have already received Y2K certification from software suppliers in the past months. We are still asking from and waiting for the others. Some vendors sent upgrade kits to make their software Y2K-compliant.

For information systems developed by ISTC, we have already identified those that need to be modified. Modifications are ongoing. Two major systems are the accounting system and the Student Records Management (SRM) system.

The accounting system will be replaced by an enterprise resource planning (ERP) system. This will initially cover accounting, purchasing, and inventory. The target completion and implementation period is second quarter of 1999 with third quart as contingency period.

The SRM (for DLSU-Manila undergraduate and graduate schools and DLSU-CSB undergraduate) is undergoing re-development using Oracle. The target completion and implementation period is third quarter of 1999 with the fourth quarter as contingency period. The SRM for La Salle Green Hills and De La Salle – Zobel are already Y2K-compliant since last year.

  Hardware (Computer clock)
 

Another concern is its effect on computer hardware – specifically the computer’s internal clock. The date and time are kept by a software called BIOS (Basic Input/Output System). This software is stored in a computer chip. Computers that are not Y2K-compliant cannot automatically change its date from December 31, 1999 to January 1, 2000.

How to assess if a computer is Y2K-compliant

Method #1:

  1. Turn on the computer and boot from MS DOS.
  2. Set the date to 12-31-1999.
  3. Set the time to 23:59:58.
  4. Wait for at least three (3) seconds for the date to change from 1999 to 2000.
  5. Display the date using the DATE command.
  6. If the date is correct (i.e., January 1, 2000), it means that the computer accepts such a date.
  7. Turn off the computer.
  8. Turn on the computer and boot from MS DOS. (Same as Step 1.)
  9. Display the date using the DATE command.
  10. If the date is correct (i.e., January 1, 2000), it means that the computer is or could be Y2K-compliant.

Method #2:

Check if the computer accepts the following dates.

  1. January 10, 2000 – first day in 2000 when the three digits are needed for month and day (i.e., 1/10/2000).
  2. February 29, 2000 – first year in a century divisible by 400 is a leap year (e.g., 1600, 2000).
  3. February 29 of other leap years beyond 2000.
  4. October 10, 2000 – first day in 2000 when four (4) digits are needed for the month and day (i.e., 10/10/2000).
  5. January 2, 2001 – first day when month/day/year (or day/month/year) order is ambiguous. That is, does 01/02/01 mean January 2, 2001, or February 1, 2001?
  6. January 2, 2034 – sometimes used as a test date format (1/2/34).

If all dates are accepted, displayed, and translated to various formats (e.g., 1/1/2003 and January 1, 2003), it means that the computer is or could be Y2K-compliant.

Solutions

Since we have a lot of computers in campus, ISTC has taken steps in identifying solutions to the Millennium Bug. Here are some of them.

  1. Manually set the date of your computer to January 1, 2000.
    (Of course, only when January 1, 2000 comes.)

    This is the simplest and cheapest solution. Actually, there is practically no cost involved.

    Most computers lower (or older) than the Pentium model are not Y2K-compliant. Note, however, that some are Y2K-compliant.

    On the other hand, there are Pentium models that are not Y2K-compliant.

    When January 1, 2000 comes, just manually set the date your computer. The computer will "accept" it and will continue to function.

    ISTC has tested this on a number of 386, 486, and Pentium models.

    If your computer is not Y2K-compliant and you do not manually set the date on January 1, 2000, the computer date will change to some other date -- e.g., January 1, 1980.

    Your files will not be deleted or corrupted. If this happens, it is not because of the Y2K Bug per se.
  2. Purchase a new computer to replace the old one.

    It is the time for the offices/units to prepare budget proposals. You may want to include any of the following:

    P27,000 - estimate cost of one complete computer set for typical office use (e.g., MS Word,
    MS Excel, MS Powerpoint, E-mail, and Internet web browsing)

    P15,000 - estimate cost of upgrade of motherboard and hard disk
    (This is sufficient for a computer with a color monitor.)

    Note: Cost is subject to change based (typically) on Philippine Peso and US Dollar exchange rate.

    In our experience, some offices insist in keeping their old units even after purchasing a new one. Please take note that old computers affect productivity. This means that it is more expensive not to upgrade/replace your old unit most especially if your unit is a 286 or 386 model.

    It is up to the unit head to determine/prioritize which computer will be upgraded/replaced first.
